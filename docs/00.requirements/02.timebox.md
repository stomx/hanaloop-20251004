# 시간 제한 (Timebox)

## 개요

이 과제는 시간 제약이 있으며, 효율적인 시간 관리와 우선순위 설정 능력도 평가 대상입니다.

## 시간 제약

### 총 소요 시간

- **집중 작업 시간**: 8-12시간
- **최대 기한**: 3일

### 의미

**8-12시간**은 실제로 코딩에 집중한 시간을 의미합니다. 생각하고, 계획하고, 문서를 읽고, 문제를 해결하는 모든 시간을 포함합니다.

**3일**은 캘린더 기준 최대 제출 기한입니다. 즉, 과제를 받은 시점부터 3일 이내에 제출해야 합니다.

## 시간 관리 전략

### 추천 일정

#### Day 1 (3-4시간)

**계획 및 기초 구축**

- [ ] 요구사항 분석 및 질문사항 정리 (30분)
- [ ] 아키텍처 설계 및 프로젝트 구조 계획 (30분)
- [ ] Next.js 프로젝트 초기 설정 (30분)
- [ ] 타입 정의 작성 (30분)
- [ ] 가짜 API 구현 (1시간)
- [ ] 시드 데이터 작성 (30분)

**마일스톤**: 데이터 레이어 완성, 기본 프로젝트 구조 확립

#### Day 2 (4-5시간)

**핵심 기능 구현**

- [ ] 레이아웃 컴포넌트 (내비게이션, 헤더) (1시간)
- [ ] UI 컴포넌트 라이브러리 구축 (1시간)
- [ ] 대시보드 홈 - 요약 카드 (1시간)
- [ ] 차트 구현 (시계열, 비교, 분포) (2시간)

**마일스톤**: 대시보드 기본 기능 작동

#### Day 3 (3-4시간)

**추가 기능 및 마무리**

- [ ] 필터링 기능 (1시간)
- [ ] 포스트 생성/수정 (1-1.5시간)
- [ ] 로딩/에러 상태 처리 (30분)
- [ ] 스타일 및 반응형 조정 (30분-1시간)
- [ ] 테스트 작성 (선택) (30분)
- [ ] 문서화 (README, 가정, 디자인 결정) (30분-1시간)

**마일스톤**: 제출 가능한 완성품

### 우선순위 가이드

#### P0 (필수 - 반드시 구현)

- 대시보드 홈 페이지
- 배출량 데이터 시각화 (최소 1-2개 차트)
- 기본 필터링 (회사 또는 날짜)
- 가짜 API (지연 + 실패 시뮬레이션)
- 로딩/에러 상태

#### P1 (중요 - 시간이 허락하면)

- 포스트 생성/수정
- 다양한 차트 (3종 이상)
- 고급 필터링 (다중 선택, 검색)
- 반응형 디자인
- 낙관적 업데이트 + 롤백

#### P2 (선택 - 시간이 남으면)

- 회사 상세 페이지
- 데이터 테이블 (정렬, 페이지네이션)
- 포스트 삭제
- 애니메이션
- 다크 모드
- 단위 테스트

## 시간 추적

### 시간 로그 작성

작업하면서 시간을 기록하면 평가자에게 투명성을 제공하고, 자신의 시간 관리 능력을 보여줄 수 있습니다.

**간단한 형식**:

```markdown
# Time Log

Total: ~10 hours

## Day 1 (2024-10-01): 4 hours
- 9:00 - 10:00: Requirements analysis & planning
- 10:00 - 11:30: Project setup, type definitions
- 14:00 - 16:30: Fake API, seed data, basic layout

## Day 2 (2024-10-02): 4.5 hours
- 10:00 - 12:00: Dashboard UI components
- 13:00 - 15:30: Charts implementation
- 16:00 - 17:00: Filters

## Day 3 (2024-10-03): 2.5 hours
- 10:00 - 11:30: Post management
- 14:00 - 15:00: Polish & documentation
```

### 추적 도구 (선택사항)

- **수동**: 메모장이나 TIME_LOG.md 파일에 직접 기록
- **자동**: Toggl, Clockify, WakaTime 등의 시간 추적 도구 사용

## 시간이 부족할 때

### 스코프 조정

시간이 부족하다고 느껴지면:

1. **우선순위 재평가**: P0 항목에 집중
2. **기능 단순화**: 복잡한 기능을 더 간단한 버전으로 축소
3. **문서로 보완**: 구현하지 못한 부분을 설계 문서나 TODO로 명시

**예시**:

```markdown
## Not Implemented (Due to Time Constraints)

### Company Detail Pages
- **Planned**: Separate pages for each company with detailed analytics
- **Current**: Inline filtering on dashboard
- **Rationale**: Focused on core dashboard experience within time limit

### Dark Mode
- **Planned**: Toggle between light/dark themes
- **Current**: Light mode only
- **Estimate**: Would need 2-3 additional hours
```

### 품질 vs 양

**10개의 반쯤 완성된 기능**보다 **5개의 잘 만들어진 기능**이 훨씬 좋습니다.

- 작동하는 코드를 우선시
- 깔끔한 코드 구조
- 명확한 타입 정의
- 기본적인 에러 처리

## 리소스 사용

### 허용되는 것

- ✅ 인터넷 검색 (Stack Overflow, MDN, 공식 문서)
- ✅ ChatGPT, GitHub Copilot 등 AI 도구 활용
- ✅ 오픈소스 라이브러리 사용
- ✅ 디자인 영감 (Dribbble, Behance 등)
- ✅ 기존 코드 샘플 참고 (이해하고 수정하여 사용)

### 금지되는 것

- ❌ 타인에게 대신 작성 요청
- ❌ 완전히 이해하지 못한 코드 복붙
- ❌ 전체 프로젝트를 다른 곳에서 복사

### 원칙

**전체적인 설계는 귀하의 고유한 창작물이어야 합니다.**

개별 유틸리티 함수나 컴포넌트를 참고하는 것은 괜찮지만, 전체 아키텍처와 디자인 결정은 스스로 해야 합니다.

## 제출 전 체크리스트

마지막 30분은 다음을 확인하는 데 사용하세요:

- [ ] 로컬에서 `npm install` → `npm run dev` 작동 확인
- [ ] `npm run build` 성공 확인
- [ ] 주요 기능 테스트 (대시보드, 필터, 포스트)
- [ ] README.md 작성 완료
- [ ] 시간 로그 업데이트
- [ ] 가정 및 트레이드오프 문서화
- [ ] Git 커밋 정리
- [ ] 불필요한 파일 제거 (.env.local, node_modules 등은 .gitignore)

## 마음가짐

### 완벽주의 피하기

**완벽한 것보다 완성된 것이 낫습니다.**

평가자들은 완벽한 솔루션을 기대하지 않습니다. 오히려 다음을 보고자 합니다:

- 시간 제약 내에서 무엇을 우선시했는가?
- 트레이드오프를 어떻게 정당화했는가?
- 제한된 시간에도 코드 품질을 유지했는가?

### 질문하기

모호한 부분이 있다면 **질문하는 것이 좋습니다**.

예시:

- "필터는 OR 로직인가요, AND 로직인가요?"
- "포스트는 회사당 여러 개 작성 가능한가요?"
- "다크모드는 필수인가요?"

질문을 통해 **비판적 사고**를 보여줄 수 있습니다.

## 타임박스의 가치

시간 제약은:

1. **실전 상황 시뮬레이션**: 실제 업무에서도 데드라인이 있습니다
2. **우선순위 결정 능력**: 무엇이 중요한지 판단하는 능력
3. **효율성**: 빠르게 학습하고 적용하는 능력
4. **의사소통**: 제약사항을 명확히 전달하는 능력

이 모든 것이 평가 대상입니다.

## 요약

- **8-12시간 집중 작업**, 최대 3일 내 제출
- **우선순위**: P0 → P1 → P2 순서로
- **품질 > 양**: 적은 기능을 잘 만드는 것이 중요
- **문서화**: 시간, 가정, 트레이드오프 기록
- **질문 환영**: 모호한 부분은 질문하세요
- **완벽보다 완성**: 작동하는 것을 우선시

시간이 부족하다면, 그것을 솔직하게 문서화하고 어떤 선택을 했는지 설명하세요. 그 자체가 평가의 일부입니다.
